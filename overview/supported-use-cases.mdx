Arcane is intended for applications that need private asset movement and a controlled compliance review path.

## Primary use cases

| Use case | What Arcane enables |
| --- | --- |
| Private card funding | Card and embedded finance products can fund card balances while preserving internal reconciliation and reducing public flow visibility |
| Private payroll flows | Payroll and earned-wage products can shield employer funding, employee allocations, and withdrawal paths |
| Regulated private transfers | Applications can submit private deposit, withdrawal, transfer, and mixed transaction shapes through supported Arcane rails |
| Case-based audit review | Auditors review approved case data instead of receiving unrestricted access to all application activity |
| Disclosure request workflow | Auditors can create disclosure requests, and application administrators can approve, close, or manage access |
| Report generation | Authorized users can generate and download transaction or case reports from scoped portal data |
| Activity logging | Disclosure requests, access, decisions, reports, and downloads are written to activity logs |
| Partner integration | Application teams can integrate APIs or SDKs so Arcane can execute, index, reconcile, and disclose private transaction activity |

## Fit assessment

Arcane is a good fit when:

- The application needs private asset movement with an API or SDK integration path.
- The organization needs a permissioned path for audit and disclosure review.
- Disclosure should be case-based, logged, and scoped to users and applications.
- The application can preserve stable product references for reconciliation.

Arcane is not a good fit when:

- The application only needs public ledger analytics.
- The review model requires every transaction detail to be public by default.
- The organization needs only a custodial wallet or identity-provider product rather than a privacy and selective-disclosure layer.
