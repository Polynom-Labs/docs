---
title: "How Arcane Works"
description: "End-to-end flow from private transaction execution to scoped audit review."
---

Arcane separates transaction execution from disclosure review.

Applications execute private transactions through Arcane APIs, SDKs, or supported privacy-pool contracts. The Auditing Portal does not sit in the user transaction path. It observes registered private transaction records, interprets encrypted audit payloads, and exposes reviewed data through permissioned workflows.

## End-to-end flow

| Step | Component | Result |
| --- | --- | --- |
| 1 | Partner application | Creates a product intent such as funding, transfer, payroll allocation, or withdrawal |
| 2 | Arcane API or SDK | Prepares private transaction inputs, proof data, encrypted outputs, and submission payloads |
| 3 | Privacy rail | Verifies proof data, updates private state, and performs public token legs when needed |
| 4 | Event or index boundary | Emits or exposes encrypted audit and transaction state |
| 5 | Arcane indexer or scanner | Reads registered activity and writes raw audit rows |
| 6 | Interpretation worker | Decrypts and normalizes audit payloads into interpreted records |
| 7 | Auditing Portal | Enforces organization, application, case, and permission scope for review and reporting |

## Disclosure model

The platform is built around scoped disclosure rather than global visibility.

| Layer | Visibility model |
| --- | --- |
| Public ledger | Contract calls, events, commitments, nullifiers, roots, and public transaction metadata when applicable |
| Encrypted audit payload | Emitted on-chain but readable only with the registered contract decoding key |
| Backend interpretation | Stored in PostgreSQL after scanner and interpretation workers process registered contracts |
| Portal access | Restricted by organization, application, case assignment, permissions, and access windows |
| Reports and logs | Generated and downloaded only through explicit report permissions and logged activity |

## Main boundaries

| Boundary | What crosses it |
| --- | --- |
| Application to Arcane | Product intent, private account, amount, asset, destination, and external reference |
| Arcane to privacy rail | Proof bytes, public signals, encrypted audit payload, relayer job, or wallet-signed transaction |
| Privacy rail to ledger | State updates, public token legs, and audit events |
| Ledger to backend | Registered events and indexed transaction state |
| Backend to portal | Scoped API responses for authenticated users |
| Portal to auditor | Case-scoped review, reports, downloads, and activity-log records |

## Operational split

| Domain | Can be used independently? | Notes |
| --- | --- | --- |
| Privacy execution | Yes | Applications can use Arcane transaction APIs or SDKs without the portal being in the user transaction path |
| Auditing Portal | Yes, with registered data sources | The portal consumes indexed and interpreted audit data; it is organized around applications, cases, reports, and user access |

The two domains are connected by the audit boundary: private execution emits or exposes encrypted records; the backend indexes and interprets them for scoped disclosure.
