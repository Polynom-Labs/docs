---
title: "Client-Side Wallet Flows"
description: "Use wallet-signed flows when end users directly authorize privacy-pool transactions."
---

Client-side wallet flows are useful when the end user should sign the privacy-pool transaction directly. This is common for non-custodial wallet applications and Stellar/Soroban integrations.

Use backend-managed wallets when your product requires server-side orchestration, card funding, payroll automation, or custodial treasury operations.

## Responsibilities

| Area | Frontend | Backend | Arcane |
| --- | --- | --- | --- |
| Wallet connection | Yes | No | No |
| User authorization | Yes | Optional verification | No |
| Private transaction preparation | SDK in browser or app | Optional helper | SDK and contract rules |
| Transaction signing | Wallet | No | No |
| Indexing and audit | No | Reads Arcane state | Yes |
| Disclosure workflows | No | Optional API integration | Yes |

## Basic flow

1. Connect the user's wallet.
2. Ask the wallet to sign a message when deriving a private address.
3. Use the SDK to prepare private transaction inputs.
4. Ask the wallet to sign and submit the transaction.
5. Track chain confirmation.
6. Let Arcane index and interpret the emitted audit event.

## Frontend safety

Client-side flows can hold user-controlled private material, but they should not receive backend-managed secrets.

Do not expose:

- Partner API keys.
- Backend signing keys.
- Relayer credentials intended for server use.
- Other users' decoded private state.
- Internal audit decoding keys.

## When to use this pattern

Use client-side wallet flows when:

- The user expects self-custody.
- The wallet is the primary trust boundary.
- The product can wait for user signature.
- Your application does not need to execute transactions automatically.

Avoid this pattern when:

- You need unattended payroll or treasury execution.
- You must link card funding to a backend ledger before completion.
- You need a controlled server-side retry and reconciliation process.
