---
title: "Reconciliation and Audit Records"
description: "Link Arcane private activity to partner product state without making all activity public."
---

Arcane integrations need two views of the same activity:

- A simple product view for users and operations.
- A permissioned review view for compliance, support, disputes, and reporting.

Reconciliation connects those views without leaking private transaction details publicly.

## Reconciliation keys

Store stable references in your backend.

| Key | Owner | Purpose |
| --- | --- | --- |
| Product object id | Partner | Card load, payroll cycle, payout, customer action, or treasury operation |
| Managed wallet reference | Partner | Links the product action to backend wallet state |
| Funding address | Partner | Explains where public funds were sent |
| Public transaction signature | Chain and partner | Confirms public funding or withdrawal |
| Shield or private transfer signature | Chain, relayer, and partner | Tracks SDK-submitted privacy-layer execution |
| Scan state | Partner | Explains what private state was indexed and when |
| Disclosure case or report id | Arcane auditing portal, when used | Supports scoped review |

## What to record

For each user-visible money movement, record:

- Product object id.
- Amount, asset, and chain.
- Managed wallet reference.
- Public funding or withdrawal signature if one exists.
- Shield, private transfer, or withdrawal signature.
- Status history.
- Retry attempts and error messages.
- Disclosure case or report reference when the audit portal is used.

## Review access

Review access should not automatically expose all private details. It should be routed through a controlled workflow.

| Actor | Typical access |
| --- | --- |
| Customer support | Limited status and product reference |
| Application administrator | Approve or deny scoped disclosure |
| Auditor | Review approved transaction fields in an active case |
| Organization administrator | Manage users, applications, and access boundaries |

See [Controlled Disclosure](/integration-guides/controlled-disclosure) for the approval path.

## Operational reports

Common reports include:

- Card funding sessions by status.
- Payroll operations by cycle or employee reference.
- Withdrawals by destination chain.
- Failed or delayed shielding operations.
- Disclosure cases and approved reports.
- Product-ledger reconciliation.

## Design guidance

- Treat product ids as your durable source of truth.
- Keep personally identifiable information out of public chain data.
- Make status updates append-only when possible.
- Use idempotency for all product ledger updates.
- Preserve enough SDK and chain evidence to explain failures.
