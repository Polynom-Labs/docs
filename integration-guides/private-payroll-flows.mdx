---
title: "Private Payroll Flows"
description: "Use Arcane for private payroll, streaming compensation, and employee payout workflows."
---

Private payroll flows let an employer or payroll platform move compensation with less public transaction leakage while preserving a controlled review path.

Use this pattern for payroll streaming, contractor payouts, earned wage access, employee withdrawals, and payroll-funded cards.

## Payroll model

```mermaid
flowchart LR
  employer["Employer treasury"]
  payroll["Payroll cycle or stream"]
  wallet["Employee managed wallet"]
  private["Private employee balance"]
  payout["Withdrawal or card funding"]

  employer --> payroll
  payroll --> wallet
  wallet --> private
  private --> payout
```

## Backend-owned records

Your backend should keep payroll concepts in your payroll system, not as Arcane-hosted objects.

| Record | Purpose |
| --- | --- |
| Payroll cycle or stream | Employer, employee, amount, asset, and schedule |
| Employer managed wallet | Backend signer and private state for employer-funded liquidity |
| Employee managed wallet | Backend signer and scan state for employee private balance |
| Operation history | Shield, private transfer, withdrawal, retry, and failure records |
| Product ledger entry | Employee-facing payroll, card, or payout status |

## Employer funding

The employer or payroll platform first funds private payroll liquidity.

1. Create or load the employer managed wallet.
2. Create a payroll funding record in your backend.
3. Detect public funding through chain RPC.
4. Shield funds with the SDK deposit flow.
5. Store the payroll cycle id, public funding signature, and shield signature.

## Employee allocation

When employees accrue wages, your backend can transfer value privately from the employer or payroll wallet to the employee wallet.

For Solana, this maps to `transactTransfer`.

Store the payroll reference next to the SDK operation:

```json
{
  "payroll_cycle_id": "payroll_2026_06",
  "employee_reference": "employee_123",
  "amount": "250.00",
  "asset": "USDC",
  "operation": "transfer"
}
```

## Employee withdrawal

Employees can move available private balance to:

- A public wallet.
- A card funding flow.
- A treasury-managed payout flow.
- Another private recipient.

Use [Withdrawals and Payouts](/integration-guides/withdrawals-and-payouts) for public payout flows and [Private Card Funding](/integration-guides/private-card-funding) for card-linked flows.

## Review path

Payroll privacy should not remove the ability to resolve regulated or operational questions.

Preserve:

- Payroll cycle and employee references.
- Public funding signatures.
- Private transfer signatures.
- Withdrawal signatures.
- Status history and retry attempts.
- Scoped disclosure or support case references when the audit portal is used.

Disclosure should be scoped by organization, application, case, time window, asset, and transaction references. See [Controlled Disclosure](/integration-guides/controlled-disclosure).

## Integration checklist

- Decide whether each employee has one managed wallet or one wallet per payroll program.
- Store payroll cycle, employee, and employer references in your backend.
- Make recurring payroll operations idempotent.
- Keep private transaction status separate from payroll product status.
- Decide how card funding and public withdrawals consume private payroll balance.
- Test the support and disclosure path before moving production payroll data.
