---
title: "Security Checklist"
description: "Secure API credentials, signing keys, private state, and disclosure workflows."
---

Review this checklist before moving production value through Arcane.

## Credentials

- Store Arcane API keys or SDK service credentials in a secret manager.
- Never expose API keys, relayer credentials, or backend signing material in frontend code.
- Use separate keys and service credentials for sandbox, staging, and production.
- Rotate keys when team access changes.
- Log administrative key changes.

## Signing and private state

- Store backend signing keys in KMS, HSM, or a managed signing service.
- Do not store raw private keys in plaintext.
- Do not store decoded UTXOs or private output caches in browser local storage.
- Restrict access to scan state and encrypted outputs.
- Use secure session cookies or equivalent backend session controls.

## Webhooks

- If hosted webhooks are enabled, verify webhook signatures.
- Store webhook or internal event ids.
- Process events idempotently.
- Use TLS for webhook endpoints.
- Alert on repeated delivery or polling failures.

## Product controls

- Validate withdrawal destinations.
- Apply velocity and amount limits.
- Require review for unusual card, payroll, or treasury activity.
- Keep customer identifiers out of public chain metadata.
- Separate customer support access from auditor access.

## Disclosure controls

- Require approval before scoped disclosure.
- Enforce organization and application boundaries.
- Use case access windows.
- Log report generation and downloads.
- Remove access when users leave the organization.
