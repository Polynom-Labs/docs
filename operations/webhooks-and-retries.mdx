---
title: "Webhooks and Retries"
description: "Process Arcane webhook events safely and idempotently."
---

Use webhooks to receive asynchronous status updates when hosted API events are enabled for your environment. For direct SDK integrations, implement the same pattern with partner-backend events and polling against the relayer, indexer, and chain RPC.

Treat webhook delivery as at-least-once and potentially out of order.

## Processing pattern

1. Verify the webhook signature when processing hosted events.
2. Store the event id.
3. Refetch the resource when the update changes money movement or product ledger state.
4. Apply the state transition idempotently.
5. Return `2xx` only after durable storage succeeds.

## Idempotency

Every create call that can move money should have an idempotency key.

```bash
Idempotency-Key: payroll_cycle_2026_06_employee_123
```

Every webhook or internal status event should also be processed once.

```text
unique(event_id)
```

## Retry domains

| Domain | Retry guidance |
| --- | --- |
| API create request | Retry with the same idempotency key |
| Webhook or internal event delivery | Store event ids and process idempotently |
| Relayer job | Retry submission only when the previous state is known |
| Chain confirmation read | Retry or switch RPC provider |
| Indexer lag | Poll resource status or wait for webhook when available |
| Product ledger update | Retry after Arcane resource is confirmed and indexed |

## Out-of-order delivery

Do not assume webhooks or internal events arrive in lifecycle order. Use the latest resource, relayer, indexer, and chain status as the source of truth.

For example, if your backend receives `funding_intent.confirmed` after `funding_intent.available`, keep the product state at `available`.
