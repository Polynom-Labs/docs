---
title: "Webhooks"
description: "Receive asynchronous status changes for Arcane resources."
---

Use webhooks to keep your backend synchronized with Arcane resource status when hosted API events are enabled. Polling should remain available as a fallback for critical status reads.

<Warning>
These webhook examples describe the intended hosted API model. If you are integrating through the SDK today, implement equivalent events in your backend and confirm hosted webhook availability with Arcane before coding against these payloads.
</Warning>

## Event shape

```json
{
  "id": "evt_...",
  "type": "funding_intent.available",
  "created_at": "2026-06-25T10:00:00Z",
  "data": {
    "object": {
      "id": "fi_...",
      "status": "available"
    }
  }
}
```

## Common events

| Event | Meaning |
| --- | --- |
| `funding_intent.funding_detected` | Public funds were observed |
| `funding_intent.available` | Private balance is indexed and usable |
| `funding_intent.failed` | Funding or shielding failed |
| `private_transfer.available` | Recipient private balance is usable |
| `private_transfer.failed` | Transfer failed |
| `withdrawal.confirmed` | Public chain confirmed withdrawal |
| `withdrawal.failed` | Withdrawal failed |
| `audit_record.created` | Audit record reference is available |

## Processing rules

- Verify the webhook signature.
- Store the event id before applying product changes.
- Process events idempotently.
- Refetch the resource from Arcane, or read the latest relayer/indexer/chain status in a direct SDK integration, before high-risk ledger changes.
- Return a `2xx` response only after durable storage succeeds.

## Retry behavior

Your endpoint should handle duplicate delivery and out-of-order arrival. Use resource status and timestamps instead of assuming webhook order.

See [Webhooks and Retries](/operations/webhooks-and-retries).
