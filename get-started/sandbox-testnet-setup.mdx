Use a sandbox or testnet environment before moving real value through Arcane.

## Environment checklist

| Requirement | Purpose |
| --- | --- |
| Arcane SDK access | Build and submit privacy-layer operations from your backend |
| Test chain RPC | Read funding transactions and submit test transactions |
| Test asset | Exercise deposit, transfer, and withdrawal flows |
| Partner backend environment | Store managed wallet state, product references, scan state, and operation history |
| Secure signing setup | Keep backend-managed keys out of frontend code |

## Suggested environments

| Rail | Use for |
| --- | --- |
| Solana devnet or test deployment | Backend-managed card funding and relayer flows |
| Stellar testnet or Futurenet | Stellar/Soroban privacy-pool flows |
| Local integration environment | Backend orchestration, state machine, and retry testing |

## Direct Solana SDK configuration

Keep Arcane, chain, and signing configuration server-side. For the current Solana SDK path, the backend needs indexer, relayer, Light RPC, prover, and Solana RPC configuration.

```bash
ARCANE_INDEXER_URL=<arcane-sandbox-indexer-url>
ARCANE_RELAYER_URL=<arcane-sandbox-relayer-url>
ARCANE_LIGHT_RPC_URL=<light-rpc-url>
ARCANE_CIRCUIT_PATH=<circuit-base-path-or-url>
SOLANA_RPC_URL=<solana-devnet-rpc-url>
```

`ARCANE_CIRCUIT_PATH` is an advanced prover setting. The SDK default is `/transaction2`, and the value is used as the base path for the `.wasm` and `.zkey` proof artifacts. See [Solana SDK](/sdks/solana-sdk).

For Stellar testing, configure the target testnet RPC and deployed contract details described in [Stellar SDK](/sdks/stellar-sdk).

## Test sequence

1. Create or restore a managed wallet row.
2. Create a partner-owned funding session.
3. Fund the returned address with test assets.
4. Shield the funds with the SDK.
5. Scan private state and read the private balance.
6. Submit a private transfer.
7. Withdraw a small amount to a public address.
8. Confirm product reconciliation and retry behavior.

## Sandbox data

Use realistic product identifiers without using production customer data.

| Field | Example |
| --- | --- |
| `card_load_id` | `card_load_test_001` |
| `customer_reference` | `test_customer_001` |
| `payroll_cycle_reference` | `test_payroll_2026_06` |
| `metadata.environment` | `sandbox` |

## Go-live gate

Move to production only after:

- Product operations and retries are idempotent.
- Funding, shielding, transfer, withdrawal, and reconciliation paths pass.
- Secrets are stored in KMS, HSM, or an equivalent managed signing layer.
- Your team can explain every status in [Status Lifecycle](/operations/status-lifecycle).
- Your compliance owner has reviewed [Controlled Disclosure](/integration-guides/controlled-disclosure).
